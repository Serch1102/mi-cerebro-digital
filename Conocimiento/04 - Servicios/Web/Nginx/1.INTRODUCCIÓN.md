## Nginx

Nginx es un ==servidor web y [[Glosario#^20bd84|proxy inverso]]== de alto rendimiento. Conocido por su ==**eficiencia**, **escalabilidad**== y capacidad para **==manejar una gran cantidad de conexiones simultáneas==**. ^55bb1a

![[nginx.png]]

  
## Estructura y configuración
Los archivos que se crean dentro de `/etc/nginx` son los siguientes directorios:![[Estructura_nginx.png]]

### Directorio de Configuración Principal
- El archivo principal de configuración se encuentra en `/etc/nginx/nginx.conf`. ==Este archivo== es fundamental ya que ==contiene la configuración global de Nginx==, lo que incluye ==ajustes relacionados con el comportamiento del servidor, los módulos cargados y las directivas de configuración generales.==


- `/etc/nginx/sites-available` y `/etc/nginx/sites-enabled`: ==Directorios utilizados para la gestión de sitios virtuales==. ==Los archivos de configuración de sitios virtuales se encuentran en `sites-available`. Para activar un sitio, se crea un enlace simbólico en `sites-enabled`.==
  
	- **`/etc/nginx/sites-available`**: Cada archivo en este directorio define la configuración específica de un sitio web, incluyendo la dirección del servidor, los archivos de acceso y error, y otras configuraciones relevantes.
	  
	- **`/etc/nginx/sites-enabled`**: Este directorio contiene enlaces simbólicos hacia los archivos de configuración de los sitios virtuales que están activos y que Nginx debe servir. ==Para activar un sitio virtual, se crea un enlace simbólico en este directorio apuntando al archivo de configuración correspondiente en `sites-available`.==
	  
	La separación de los directorios de sitios disponibles y activados proporciona una forma conveniente de habilitar y deshabilitar sitios web sin necesidad de modificar directamente los archivos de configuración principales. Este enfoque facilita la gestión y organización de múltiples sitios web en un servidor Nginx.

### Configuración Básica del Servidor en Nginx

Una de las características principales de nginx es que ==desde el archivo `nginx.conf`, puedes definir servidores utilizando bloques `server { }`.== Esto se usa para ==definir la configuración para un servidor web específico== o un "sitio virtual".

==Cada bloque `server` se emplea para especificar cómo Nginx debe responder a las solicitudes entrantes== basándose en diferentes criterios, como el nombre del servidor, el puerto o la dirección IP. A continuación, se muestra un ejemplo de configuración básica:

```nginx
server {
    listen 80;
    server_name ejemplo.com www.ejemplo.com;

    location / {
        root /ruta/a/tu/sitio;
        index index.html;
    }
}
```

En este ejemplo, se define un servidor que ==escucha en el puerto 80== y ==responde a las solicitudes para los nombres de servidor `ejemplo.com` y `www.ejemplo.com`==. La directiva ==`listen` especifica el puerto en el que el servidor escucha las solicitudes entrantes==.

Dentro del bloque `location /`, se configura la raíz del sitio web (`root`) y se especifica el nombre del archivo de índice (`index.html`). Esto significa que al acceder al dominio a través del puerto 80, Nginx servirá el archivo `index.html` ubicado en la ruta especificada como la raíz del sitio.


---
Ejemplo:
```nginx
# Definición del servidor
server {
    # Escucha en el puerto 80
    listen 80;
    
    # Nombres de dominio que el servidor debe responder
    server_name ejemplo.com www.ejemplo.com;

    # Configuración de la raíz del sitio y el archivo de índice
    location / {
        root /var/www/html;
        index index.html;
    }

    # Configuración adicional si es necesario
}
```

En este ejemplo:
- La raíz del sitio se establece en `/var/www/html`, lo que significa que los archivos del sitio se esperan en ese directorio.
- El archivo de índice predeterminado es `index.html`, lo que significa que cuando los usuarios accedan al sitio, se mostrará este archivo si no se especifica ningún otro recurso en la URL.
---

Esta configuración básica del servidor en Nginx proporciona una base sólida para hospedar sitios web y gestionar solicitudes entrantes de manera efectiva.

---

### Registro y acceso

Como cualquier servidor web y, especialmente, los que tienen funciones de proxy tienen funciones de registro:

- Los errores se registran en `/var/log/nginx/error.log`.
  Estos registros son cruciales para identificar problemas de configuración, errores de aplicación y otras situaciones que requieran atención.

- Los registros de accesos se encuentran en `/var/log/nginx/access.log`.
  Proporcionan detalles sobre las solicitudes entrantes, como direcciones IP, URLs solicitadas, códigos de estado HTTP y otros datos relevantes.

## Parámetros de conexión del servidor

### Configuración de Workers en Nginx

==Para aprovechar al máximo los recursos disponibles en el servidor, especialmente los núcleos de la CPU, es recomendable configurar adecuadamente el número de workers en Nginx==. 

==En el archivo principal de configuración de Nginx==, generalmente llamado `nginx.conf`, ==puedes especificar la cantidad de workers que Nginx== debe utilizar. ==La configuración más común es utilizar el valor `auto`==, lo que permite que ==Nginx determine automáticamente la cantidad óptima de workers== según la cantidad de núcleos de la CPU disponibles en el servidor.

Aquí está la configuración en el archivo `nginx.conf`:

```nginx
worker_processes auto;
```

 Esto ==garantiza una distribución eficiente de la carga y una mejor utilización de los recursos del sistema==.

Es importante tener en cuenta que ==el número de workers no debe exceder el número de núcleos de CPU disponibles, ya que esto puede causar sobrecarga y degradar el rendimiento del servidor.==


### Puertos en Nginx

Dentro de un bloque `server` en la configuración de Nginx, puedes especificar los puertos en los que el servidor debe escuchar las solicitudes entrantes. Aquí te muestro cómo hacerlo:

```nginx
# Para escuchar en el puerto 80 como servidor predeterminado
listen 80 default_server;

# Para escuchar en el puerto 80 para conexiones IPv6 solamente
listen [::]:80 default_server ipv6only=on;
```

En el primer ejemplo, Nginx escucha en el puerto 80 y marca este servidor como predeterminado para las solicitudes entrantes que no coinciden con ningún otro servidor configurado.

En el segundo ejemplo, Nginx escucha en el puerto 80 para conexiones IPv6 solamente y habilita la opción `ipv6only=on` para restringir las conexiones solo a IPv6.

### Timeouts en Nginx

==Los timeouts son parámetros importantes en la configuración de Nginx y su ajuste depende del rol que tenga Nginx en tu arquitectura, ya sea como proxy, servidor de archivos estáticos u otro.==

Aquí están algunos de los parámetros de timeouts que puedes ajustar en la configuración de Nginx:

```nginx
keepalive_timeout 10;         # Tiempo máximo que se mantienen activas las conexiones keepalive (por defecto 75)
keepalive_requests 1024;      # Número máximo de solicitudes que se pueden enviar a través de una conexión keepalive (por defecto 100)
client_header_timeout 10;     # Tiempo de espera para recibir la cabecera del cliente (por defecto 60)
client_body_timeout 10;       # Tiempo de espera para recibir el cuerpo de la solicitud del cliente (por defecto 60)
send_timeout 10;              # Tiempo de espera para enviar datos al cliente (por defecto 60)
sendfile on;                  # Habilita la directiva sendfile para enviar archivos estáticos de manera eficiente (por defecto desactivado)
tcp_nopush on;                # Habilita la directiva tcp_nopush para enviar datos en segmentos (por defecto desactivado)
tcp_nodelay on;               # Habilita la directiva tcp_nodelay para desactivar el algoritmo de Nagle (por defecto activado)
```

Estos parámetros te permiten ajustar los tiempos de espera y la gestión de conexiones en Nginx para optimizar el rendimiento y la eficiencia del servidor web. Ten en cuenta que estos son solo algunos de los parámetros disponibles, y puedes ajustarlos según tus necesidades específicas y el entorno de tu aplicación.


### Configuración de Logs en Nginx

Dentro del archivo de configuración básica de `nginx.conf`, puedes definir la configuración de logs para registrar errores y accesos. Aquí está cómo se realiza:

```nginx
http {
    # Otras configuraciones de HTTP aquí...

    # Archivo de registro de errores predeterminado
    # (esto se utiliza solo cuando no se sobrescribe error_log a nivel de server{})
    error_log  logs/error.log warn;

    # Formato a utilizar en los archivos de log
    log_format  main  '$remote_addr - $remote_user [$time_local] "$request" '
                      '$status $body_bytes_sent "$http_referer" '
                      '"$http_user_agent" "$http_x_forwarded_for"';

    # Archivo de registro de acceso predeterminado
    # (esto se utiliza solo cuando no se sobrescribe access_log a nivel de server{})
    access_log logs/access.log main;

    # Resto de la configuración...

    server {
        # Configuraciones del servidor...

        location / {
            # Configuraciones de la ubicación...
        }
    }
}
```

En este fragmento de configuración:

- `error_log`: ==Especifica la ubicación del archivo de registro de errores==. En este caso, los errores se registran en el archivo `logs/error.log` con un nivel de advertencia (`warn`). Este archivo se utiliza cuando no se define un `error_log` específico a nivel de servidor.
- `log_format`: ==Define el formato del registro de acceso.== En el ejemplo, se utiliza un formato llamado `main` que ==incluye información detallada sobre la solicitud, como la dirección IP del cliente, la fecha y hora, la solicitud HTTP, el código de estado, el tamaño de la respuesta y más==.
- `access_log`: ==Especifica la ubicación del archivo de registro de acceso.== Aquí, los accesos se registran en el archivo `logs/access.log` utilizando el formato `main` definido anteriormente.

### Módulos en Nginx

==Nginx ofrece una amplia variedad de módulos que permiten ampliar su funcionalidad==. Algunos de los módulos más importantes incluyen:

- **HTTP Referer**: Filtra las peticiones recibidas en función de la cabecera Referer.
- **HTTP Limit Zone**: ==Limita el número de conexiones simultáneas desde un mismo cliente.==
- **User ID**: Proporciona cookies identificativas.
- **FLV**: Permite reproducir vídeo en streaming.
- **Perl**: Permite ejecutar código Perl directamente dentro de Nginx.
- **WebDAV**: Ofrece soporte para el protocolo WebDAV.
- **Secure Link**: Permite proteger páginas mediante una clave secreta.
- **XSLT**: Proporciona funcionalidad para el post-procesamiento de páginas mediante XSLT.
---
#### Ejemplo:

Supongamos que deseamos filtrar las solicitudes entrantes basadas en la cabecera Referer y bloquear aquellas que no provengan de nuestro propio dominio.

```nginx
http {
    # Configuración de HTTP

    server {
        listen 80;
        server_name mi_sitio.com;

        # Configuración del módulo HTTP Referer
        location / {
            valid_referers none blocked mi_sitio.com;
            if ($invalid_referer) {
                return 403; # Devuelve un error 403 Forbidden para solicitudes con Referer no válido
            }

            # Resto de la configuración para el sitio
        }
    }
}
```

En este ejemplo:

- `valid_referers`: Especifica los referentes válidos para las solicitudes. En este caso, se permiten referentes de nuestro propio dominio (`mi_sitio.com`), así como los referentes nulos y bloqueados.
- `$invalid_referer`: Es una variable interna de Nginx que se establece en `true` si el referente de la solicitud no está en la lista de referentes válidos.
- `return 403`: Devuelve un código de estado HTTP 403 Forbidden para las solicitudes con un referente no válido, bloqueando así el acceso a recursos protegidos.

Este es solo un ejemplo simple de cómo se puede utilizar el módulo HTTP Referer en Nginx para controlar el acceso a recursos basados en la cabecera Referer de las solicitudes HTTP. Los módulos en Nginx proporcionan una gran flexibilidad para adaptar el comportamiento del servidor según las necesidades específicas de cada aplicación o sitio web.

---

Estos módulos permiten adaptar y extender las capacidades de Nginx para satisfacer las necesidades específicas de diferentes aplicaciones y escenarios de uso.
